# 认证体系重设计需求稿（交付设计师）

## 1. 文档目的
本需求稿用于指导独立设计师完成一套全新的认证原型，用于与现有方案做对比评审。  
设计师仅凭本稿即可完成完整设计，不依赖额外口头说明。

## 2. 项目背景
当前前端认证体验存在以下问题：
1. 信息架构分散，登录、注册、验证、重置同屏堆叠导致认知负担高。
2. 视觉层级弱，品牌识别度不足，缺乏“专业可信”的第一印象。
3. 关键动作路径不聚焦，主次操作权重接近。

本次任务是“重做”，不是“微调”。  
不要沿用现有页面布局和视觉风格，只保留业务必需流程和接口契约。

## 3. 目标与成功标准
### 3.1 目标
1. 显著提升品牌感与专业度，做到第一眼即与旧版拉开代差。
2. 统一租户端与管理端认证视觉语言。
3. 保持可开发落地性，不要求后端接口改造。

### 3.2 成功标准
1. 评审者在 5 秒内可识别主路径和主操作。
2. 登录、注册、邮箱验证、忘记密码路径清晰可走通。
3. 双端和双主题下层级一致，不出现信息缺失。

## 4. 范围与边界
### 4.1 范围内
1. 租户认证主页：同一卡片切换 `登录/注册`，默认打开 `登录`。
2. 租户邮箱验证页：独立页面。
3. 租户忘记密码页：独立页面，抽屉式分步。
4. 管理员登录页：与租户统一视觉体系，但仅保留登录。
5. 桌面端与移动端。
6. 浅色与深色双主题。

### 4.2 范围外
1. 不接入真实第三方登录，仅保留视觉占位入口。
2. 不改后端 API 与字段。
3. 不包含登录后业务后台页面设计。

## 5. 关键产品决策（已锁定）
1. 认证主页采用左右分栏叙事结构，Desktop 推荐 `40/60` 比例。
2. 视觉方向为“高级极简”，强调留白、克制、专业。
3. 字体策略为“双字体”，标题与正文区分。
4. 文案策略先中文，后续再映射多语言 key。
5. 注册页保留并必填 `租户名称`。
6. 注册页新增 `确认密码` 字段。
7. `忘记密码` 保留，并独立为步骤页。
8. `邮箱验证` 独立为步骤页。
9. 管理端采用同版式精简版，不开放注册。
10. 动效强度为高，但需提供 reduced-motion 兜底。

## 6. 用户与场景
1. 租户用户：首次注册、回访登录、忘记密码、邮箱验证激活。
2. 管理员用户：高频登录控制台，关注安全与稳定。
3. 核心场景：用户在最短路径内完成身份认证并进入系统。

## 7. 信息架构
### 7.1 页面结构
1. 认证主页 `Tenant Auth Landing`
2. 邮箱验证页 `Tenant Verify`
3. 忘记密码页 `Tenant Forgot`
4. 管理员登录页 `Admin Login`

### 7.2 路径关系
1. 认证主页默认登录态。
2. 登录态可进入忘记密码页。
3. 注册成功后进入邮箱验证页。
4. 验证成功后返回登录态并可继续登录。

## 8. 页面规格（必须交付 16 张画板）
请按以下命名交付，命名不可变更：

1. `TenantAuth-Desktop-Light`
2. `TenantAuth-Desktop-Dark`
3. `TenantAuth-Mobile-Light`
4. `TenantAuth-Mobile-Dark`
5. `TenantVerify-Desktop-Light`
6. `TenantVerify-Desktop-Dark`
7. `TenantVerify-Mobile-Light`
8. `TenantVerify-Mobile-Dark`
9. `TenantForgot-Desktop-Light`
10. `TenantForgot-Desktop-Dark`
11. `TenantForgot-Mobile-Light`
12. `TenantForgot-Mobile-Dark`
13. `AdminLogin-Desktop-Light`
14. `AdminLogin-Desktop-Dark`
15. `AdminLogin-Mobile-Light`
16. `AdminLogin-Mobile-Dark`

每张画板必须包含：
1. 一个主状态。
2. 至少一个反馈状态（错误、成功、提示、加载任一）。
3. 动效注释（触发条件、时长、缓动、目标状态）。

## 9. 页面内容要求
### 9.1 Tenant Auth Landing（登录/注册同卡片）
1. 左侧叙事区：
1. 产品标识位。
2. 一句价值主张。
3. 三条能力要点（稳定、安全、可观测/可控）。
2. 右侧认证卡片：
1. 顶部切换器：`登录 | 注册`。
2. 默认登录激活。
3. 登录态字段：邮箱、密码、忘记密码入口、主按钮。
4. 注册态字段：租户名称、邮箱、密码、确认密码、主按钮。
5. 第三方登录入口：占位展示并标记“即将支持”。

### 9.2 Tenant Verify（独立页）
1. 字段：邮箱、验证码。
2. 主按钮：完成验证。
3. 辅助操作：重新发送验证码、返回登录。

### 9.3 Tenant Forgot（独立页，抽屉式分步）
1. Step 1：输入邮箱并发送验证码。
2. Step 2：抽屉上滑后输入验证码与新密码并提交重置。
3. 页面中需可视化表达“抽屉式分步机制”。

### 9.4 Admin Login（精简版）
1. 与租户端同视觉语言。
2. 仅保留管理员登录表单，不出现注册入口。
3. 字段：用户名、密码。

## 10. 视觉规范要求
1. 整体风格：高级极简、专业可信、强留白。
2. 禁止：
1. 套用当前旧版样式。
2. 过重装饰、廉价渐变、模板化 SaaS 套壳风。
3. 字体：
1. 标题与正文使用双字体策略。
2. 标题字体需有品牌特征，正文保持高可读。
4. 色彩：
1. 浅色与深色都需保证对比度和层级清晰。
2. 避免“纯黑/纯白硬切割”，采用专业中性色体系。
5. 组件：
1. 输入框、按钮、标签、提示反馈保持统一规则。
2. 主操作必须比次级操作更高视觉权重。

## 11. 动效规范要求（需在画板注释中写明）
1. 登录/注册滑块切换：`380ms`，`ease-out`。
2. 卡片入场：`360ms`，轻弹性或近似 spring。
3. 忘记密码抽屉上滑：`420ms`，`ease-in-out`。
4. 邮箱验证成功转场：`300ms`。
5. 管理端登录成功转场：`260ms`。
6. 移动端首屏入场：`280ms`。
7. `prefers-reduced-motion`：禁用位移动效，保留淡入或直接切换。

## 12. 文案与语言规范
1. 本轮原型主文案使用中文。
2. 文案语气：简洁、可信、不过度营销。
3. 不直接展示技术错误码或内部术语作为主文案。
4. 提示与错误信息要可理解且可操作。

## 13. 开发可落地约束（必须遵守）
1. 不改现有后端错误契约：`{ error: { code, message } }`。
2. 注册字段保留 `tenant_name` 且视为必填。
3. 不要求新增后端接口。
4. 所有新增 UI 需可映射到现有前端认证流程。

## 14. 无障碍与可用性要求
1. 文本与背景对比度满足 WCAG AA。
2. 所有表单项有清晰标签与错误提示位置。
3. 主次按钮在视觉和键盘焦点上可区分。
4. 移动端单手操作区域优先保证主按钮可达。

## 15. 交付物格式
1. 原型源文件：Pencil 或 Figma（任选其一）。
2. 16 张画板全部齐全，命名严格一致。
3. 每张画板含动效注释与状态注释。
4. 导出预览图：每张画板 1 张 PNG。
5. 一页说明：字体、颜色、间距、圆角、按钮层级规则。

## 16. 评审打分维度（用于对比）
1. 品牌感与专业度提升幅度。
2. 信息层级与主路径清晰度。
3. 登录/注册/验证/重置流程可理解性。
4. 双端与双主题一致性。
5. 可开发落地性与实现风险。

## 17. 验收清单
1. 16 张画板是否齐全且命名正确。
2. 是否完整覆盖四类页面与关键状态。
3. 是否满足“登录/注册同卡片、默认登录、保留忘记密码”。
4. 是否体现“忘记密码抽屉式分步”。
5. 是否统一租户端与管理端视觉语言。
6. 是否提供高动效规范和 reduced-motion 兜底。
7. 是否可在不改后端接口前提下落地。
